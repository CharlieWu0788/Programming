Welcome to Tetris Battle!

To defeat the dastardly Z-block Zeke, you must fill rows of the game board with your falling tetromino blocks.

---Usage---

This program can be run using the Jack Compiler and VM Emulator on the Nand2Tetris website. Note, however, that it may take a minute or two to compile due to the size of the sprites.

---Controls---

Left arrow: moves tetromino left
Right arrow: moves tetromino right
Down arrow: moves tetromino down at a greater rate than normal
Up arrow: rotates tetromino
Space bar: immediately drops tetromino as far down as it can go

---Classes Overview---

Main: creates, runs, and disposes of a Tetris game.
Tetris: runs the main game loop, handling most of the program's logic and timing.
DisplayGrid: manages the grid on which gameplay occurs, drawing out the location of tetrominoes, locked blocks, and ghost tetrominoes (i.e., the outlines of where a tetromino will land).
Tetromino: contains the data for each tetromino object (including sprites, coordinates, and current rotation).
Enemy: contains the data for Z-block Zeke (including his coordinates and animation parameters).
Sprites: a container for the massive sprite-drawing functions generated by Nand2Tetris' Bitmap Editor tool. (Also includes additive and subtractive poking methods so that partially-transparent sprites can be drawn without destroying their backgrounds).

---Tricky Sections of the Code---

DisplayGrid's "clearFilledLines" method:
This thing is a behemoth! The first section detects filled lines and uses that information to build the "lineOffset" array. The lineOffset array is essentially just a way to keep track of how far down each row will need to move in order to successfully fill in the gaps left by the cleared lines. The next section puts that array to use, changing each block's value according to its row's offset amount. If, for example, row 8 has an offset of 2 (i.e., lineOffset[8] == 2), then its new values will be equal to the current values of row 6 (the row 2 blocks above it). Following the same procedure, row 6 will be set to the values of row 4. But what of row 0? Since there are no negative rows to draw data from, rows too close to the top of the grid will simply be filled with zeroes.

Tetris' "rotateAndKick" method:
Another rather unsightly contrivance. This method facilitates wall-kicking and floor-kicking behaviors which prevent tetrominoes from getting stuck in walls or locked blocks when they rotate. In order to do this, upward, rightward, and leftward offsets must be tested. If one of these is able to place the tetromino in a legal position, then it is adopted! Otherwise, the rotation is cancelled altogether. However, there's one problem. The line block can become doubly stuck because one of the sides of its sprite includes two empty rows, allowing it to sidle up close to a wall then rotate two blocks into it. In order to address this, another set of offsets must be tested, this time attempting to move the tetromino two blocks leftwards, rightwards, or upwards. Once again, if none of these offsets can place the tetromino in a legal position, the rotation maneuver is cancelled!
